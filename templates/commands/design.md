## Outline

Goal: Analyze the active feature specification to identify all UI components, screens, and interactions, then generate visual mockups (wireframes + HTML/CSS prototypes) in a `mockups/` directory within the feature folder.

This command combines two concerns into one step:
1. **Component Analysis** — extract screens, UI elements, states, and relationships from the spec
2. **Visual Mockup Generation** — produce wireframe-quality HTML/CSS prototypes for each identified screen

This is an **optional** step. If the user skips it, the workflow proceeds directly from `/speckit.specify` (or `/speckit.clarify`) to `/speckit.plan`.

Execution steps:

1. Run `{SCRIPT}` from repo root **once**. Parse JSON payload fields:
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   For single quotes in args like "I'm Groot", use escape syntax: e.g 'I'\''m Groot' (or double-quote if possible: "I'm Groot").

2. **Read the spec**: Load `FEATURE_SPEC` (spec.md) completely.

3. **Component Analysis** — Extract from the spec:

   a. **Screens / Pages**: Identify every distinct screen or view the user interacts with.
      - Name each screen descriptively (e.g., "Dashboard", "User Profile", "Settings")
      - Note the primary purpose of each screen

   b. **UI Components per Screen**: For each screen, list:
      - Interactive elements (buttons, forms, inputs, dropdowns, toggles)
      - Display elements (cards, lists, tables, charts, media)
      - Navigation elements (menus, tabs, breadcrumbs, links)
      - Feedback elements (modals, toasts, alerts, loaders)

   c. **Component States**: For interactive components, identify:
      - Default, hover, active, disabled, loading, error, empty states

   d. **Screen Flow**: Map how users navigate between screens:
      - Entry points (where does the user start?)
      - Transitions (what action leads to which screen?)
      - Back/cancel flows

   e. **Shared Components**: Identify components reused across multiple screens (headers, footers, navigation bars, common buttons)

4. **Create mockups directory structure**:

   ```
   FEATURE_DIR/
     mockups/
       components.md
       wireframes/
         .gitkeep
       prototypes/
         index.html
         styles.css
   ```

5. **Write `components.md`** with the following structure:

   ```markdown
   # Component Analysis: [FEATURE NAME]

   **Source**: [Link to spec.md]
   **Created**: [DATE]

   ## Screens

   ### [Screen Name]
   - **Purpose**: [One-line description]
   - **Entry Point**: [How user arrives here]
   - **Components**:
     - [Component] — [role] — States: [list states]
   - **Actions**: [What user can do here]
   - **Navigates to**: [Connected screens]

   (repeat for each screen)

   ## Shared Components

   | Component | Used In | Description |
   |-----------|---------|-------------|
   | [Name]    | [Screens] | [What it does] |

   ## Screen Flow

   [Screen A] → [action] → [Screen B]
   [Screen B] → [action] → [Screen C]
   ...
   ```

6. **Generate HTML/CSS prototypes** in `prototypes/`:

   a. **`index.html`**: Create a single-page wireframe that includes:
      - All identified screens as sections (use `<section id="screen-name">`)
      - Basic layout structure matching the component analysis
      - Placeholder content for all UI components
      - Navigation links between screen sections
      - Semantic HTML5 elements (header, nav, main, section, article, footer)
      - No external dependencies — pure HTML only
      - A brief comment at the top: `<!-- Wireframe prototype generated by /speckit.design -->`

   b. **`styles.css`**: Create wireframe-style CSS:
      - Monochrome color scheme (grays, whites, light borders) — wireframe aesthetic
      - Box-based layouts to clearly show component boundaries
      - Placeholder styling for images/media (gray boxes with labels)
      - Responsive basic grid or flexbox layout
      - Visible component labels/annotations
      - Typography: system fonts only, clear hierarchy
      - No animations, no decorative styling — structure only
      - A brief comment at the top: `/* Wireframe styles generated by /speckit.design */`

   c. **Design constraints for prototypes**:
      - Wireframe fidelity: LOW — show structure and layout, not visual design
      - Every interactive element should be visible but not functional
      - Use border/outline to make component boundaries obvious
      - Label each section/component with its name from components.md
      - Include placeholder text: "Lorem ipsum" or descriptive placeholders like "[User Name]", "[Profile Image]"

7. **Create `wireframes/.gitkeep`**: This directory is reserved for externally generated wireframe images (e.g., from Pencil, Figma exports, or future API integrations). Leave it empty with a `.gitkeep` file.

8. **Report completion**: Summarize what was generated:
   - Number of screens identified
   - Number of components identified
   - Number of shared components
   - List of generated files
   - Recommend the user to review `prototypes/index.html` in a browser
   - Suggest next step: `/speckit.plan`

## Operating Constraints

- **Read-only on spec.md**: Do NOT modify the spec file. This command only reads it.
- **No implementation decisions**: Do not choose frameworks, libraries, or tech stacks. This is about visual structure, not technology.
- **Wireframe fidelity**: Keep prototypes LOW fidelity. Gray boxes, outlines, labels. No polish.
- **Self-contained prototypes**: HTML/CSS must work by opening `index.html` directly in a browser. No build tools, no dependencies.
- **Component naming**: Use names from the spec's user stories and requirements. Don't invent new terminology.

## Key rules

- Use absolute paths
- Do NOT modify spec.md — this is a read-only analysis step
- Prototypes must be self-contained (no external dependencies)
- Keep wireframe fidelity LOW — structure over aesthetics
- If the spec has no clear UI components (e.g., pure backend/library feature), inform the user and skip mockup generation gracefully
